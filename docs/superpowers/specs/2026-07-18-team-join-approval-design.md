# 团队加入审批设计

## 背景

当前团队邀请码加入接口会直接把申请人写入 `teams.member_ids`，导致任何持有邀请码的用户都能立即进入团队。本需求将加入流程改为管理员审批制，同时让申请人能看到申请状态，并支持被拒绝后再次申请。

## 目标与范围

- 新用户通过邀请码申请加入团队后，必须由团队 owner 或 admin 审批。
- 申请人可查询自己的申请状态：待审批、已通过或已拒绝。
- 管理员可在现有团队详情的成员管理区域批准或拒绝申请。
- 被拒绝的申请可以再次提交；重复的待审批申请不会创建重复记录。
- 保持现有团队角色、成员权限和邀请码格式不变。

## 数据模型

新增 `team_join_requests` 表：

- `id`：主键。
- `team_id`：关联团队。
- `user_id`：关联申请用户。
- `status`：`pending`、`approved` 或 `rejected`。
- `created_at`、`updated_at`：申请和状态更新时间。

对 `team_id + user_id` 建立唯一约束，使用同一条记录表示当前申请状态，不要求保留每次申请的完整历史；拒绝后重新申请时将状态重置为 `pending` 并更新 `updated_at`，不产生重复待审批记录。数据库初始化增加建表语句，兼容已有数据库的迁移启动流程。

## 接口与权限

### 提交加入申请

`POST /api/teams/:id/join`

- 未登录、团队不存在、用户已是成员时返回明确错误。
- 无申请记录：创建 `pending` 申请，返回待审批状态。
- 当前为 `pending`：返回审核中，不修改记录。
- 当前为 `rejected`：重置为 `pending`，允许再次申请。
- 不再直接修改 `teams.member_ids`。

### 查询个人申请状态

`GET /api/teams/:id/join-status`

仅允许当前登录用户查询自己的状态，非团队成员也可访问。返回团队存在性和当前状态；无记录时返回 `none`，用于前端区分首次申请、审核中和可再次申请。

### 获取团队详情

`GET /api/teams/:id`

保留现有成员和团队信息。仅当当前用户角色为 owner/admin 时附加 `joinRequests`，包含申请人的 id、邮箱、显示名、头像、状态和更新时间；普通成员不返回申请列表。

### 审批申请

复用 `PUT /api/teams/:id/members`，新增动作 `approveJoin` 和 `rejectJoin`：

- 仅 owner/admin 可操作；admin 遵循现有成员管理权限边界。
- 批准时校验申请仍为 `pending`，幂等加入 `member_ids`，并将申请置为 `approved`。
- 拒绝时将申请置为 `rejected`，不改变成员列表。
- 申请不存在、已处理或目标用户/团队无效时返回明确错误。

## 前端交互

- 加入团队弹窗提交后根据响应显示“申请已提交，等待管理员审批”“已在审核中”或“申请已被拒绝，可再次申请”。
- 再次输入邀请码时先查询申请状态，待审批状态不重复提交；拒绝状态允许重新提交。
- 团队详情的成员管理弹窗增加“待审批申请”区块，仅 owner/admin 可见；每条申请显示用户基本资料、申请时间和批准/拒绝按钮。
- 审批操作完成后刷新团队详情；批准后申请人在下一次状态查询/团队列表刷新中看到已加入团队。

## 错误处理与一致性

- 所有新增接口沿用现有认证中间件和 JSON 错误格式。
- 批准操作应在事务或等价的一致性处理内完成申请状态与成员列表变更，避免出现已批准但未加入成员列表的中间状态。
- 团队删除后申请记录通过删除清理或团队不存在校验失效，不能继续审批。
- 普通成员、非成员和未登录用户不能读取他人申请或执行审批。

## 验证计划

- 数据库初始化和旧数据库迁移可重复执行。
- API 场景：首次申请、重复申请、拒绝后再次申请、批准、拒绝、重复审批、无权限和团队不存在。
- 前端场景：申请状态文案、待审批列表展示、批准/拒绝后的刷新和已批准成员可见性。
- 运行项目现有 lint/build，并执行针对团队 API/组合式函数的可用测试；不修改与本需求无关的失败项。

## 非目标

- 不新增邮件、站内信或推送通知。
- 不改变邀请码生成和解析规则。
- 不重构现有团队成员权限模型。
