# 登录失效提示与脚本集市命名设计

## 目标

当用户的登录态在 API 请求期间失效时，显示一个全局对话框，让用户主动返回登录页；同时将面向用户的“插件集市”统一命名为“脚本集市”。

## 范围

- 所有前端 API 请求收到 HTTP 401 时，进入同一套登录失效处理。
- 对话框文案为“登录已失效，请重新登录”，提供“返回登录”按钮。
- 对话框显示期间不自动导航；用户点击按钮后才清理本地会话并转到 `/login`。
- 并发或连续的多个 401 只显示一个对话框。
- 更新工作区导航、集市列表、详情、提交页的可见文案与页面标题为“脚本集市”。

## 非目标

- 不修改 `/workspace/marketplace` URL、接口路径、数据模型或数据库字段。
- 不改变正常登出流程、登录页流程或 HTTP 401 以外的错误处理。

## 架构与数据流

新增一个客户端全局认证失效状态与处理函数，由请求封装在收到 401 时调用。该状态由根级对话框组件订阅并控制显示。

1. 任一 API 请求返回 401。
2. 统一请求封装触发认证失效状态；若对话框已打开则不重复触发。
3. 根级对话框显示登录失效提示。
4. 用户点击“返回登录”。
5. 处理函数清除本地 session、token 与认证提示 cookie，并导航至 `/login`。

现有未使用统一封装的 composable 和页面请求会接入同一请求处理入口，确保脚本集市与其余受保护功能获得一致行为。

## 错误处理

- 非 401 响应沿用页面现有错误提示。
- 解析失败的 401 仍应触发失效提示，不能依赖响应 JSON。
- 登录页的登录、注册、找回密码等请求不应因为自身返回的 401 触发失效对话框。

## 验证

- 模拟失效 token 请求脚本集市接口，确认仅显示一个对话框且不自动跳转。
- 点击“返回登录”后确认清理本地认证信息并进入登录页。
- 检查导航、列表页、详情页、提交页及浏览器标题均显示“脚本集市”。
- 运行项目构建，确保类型检查与打包通过。
